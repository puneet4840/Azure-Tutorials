# Azure Traffic Manager Components and Routing Rules

Traffic Manager ke kuch core components hote hain jo milkar decide karte hain ki user ko kis server ya endpoint par bhejna hai.
- Traffic Manager Profile.
- Endpoints.
- Routing Method.
- Health Monitoring.
- DNS Configuration.
- TTL.
- Traffic Manager Name Servers

Ab ek ek karke components ko samjhte hain.

<br>
<br>

### Traffic Manager Profile

Traffic Manager Profile ek configuration container hota hai jisme saari settings store hoti hain. 

Traffic Manager Profile mein saari configuration store hoti hai.

Profile ek policy configuration object hai jisme decide hota hai ki traffic kaise distribute hoga.

Isme mainly ye cheeze configure hoti hain:
- Routing method.
- Endpoints.
- Health checks.
- DNS settings.
- TTL.

Ye sab milkar decide karte hain - “User ko kis endpoint par bhejna hai”.

<br>

**Real Example**:

Maan lo ek company ki website hai:
```
www.myshop.com
```

Aur uske servers 3 jagah hain:
- East US.
- West Europe.
- Southeast Asia.

Ab company chahti hai:
- US users → East US.
- Europe users → West Europe.
- Asia users → Southeast Asia

Toh iske liye ek Traffic Manager Profile banaya jayega.

Example:
```
Traffic Manager Profile Name: myshop-traffic-profile
Routing Method: Performance
DNS Name: myshop.trafficmanager.net
```

Ab jab user request karega:
```
www.myshop.com
```

DNS usko redirect karega:
```
myshop.trafficmanager.net
```

Aur Traffic Manager decide karega:
```
User ko kaunsa server milega
```

<br>
<br>

### Endpoints

Endpoints wo actual destinations hote hain jahan Traffic Manager user ko bhejta hai.

Endpoint = Actual application server ya service.

Ye kuch bhi ho sakta hai:
- Azure Web App.
- Azure VM.
- Azure Load Balancer.
- Azure App Gateway.
- External server.
- On-premise datacenter.

<br>

**Endpoint ke Types**:

Traffic Manager mein 3 type ke endpoints hote hain.

**Azure Endpoints**:

Ye endpoints Azure ke andar wali services hoti hain.

Examples:
- Azure Web App.
- Azure VM.
- Azure Load Balancer.
- Azure App Service

Example:
```
Traffic Manager
      |
      |
-------------------------
|          |            |
WebApp1    WebApp2      WebApp3
(East US)  (Europe)     (Asia)
```

Real Example:

Company ki website Azure Web Apps par host hai.
```
webapp-eastus.azurewebsites.net
webapp-europe.azurewebsites.net
webapp-asia.azurewebsites.net
```
Traffic Manager inhe endpoints bana deta hai.

<br>

**External Endpoints**:

External endpoints Azure ke bahar wale servers hote hain.

Example:
- AWS servers.
- On-premise datacenter.
- Other cloud provider.

Example:
```
Server1 → AWS
Server2 → Azure
Server3 → On-premise
```
Traffic Manager multi-cloud routing bhi kar sakta hai.

Real Example:

Company ka setup:
```
Azure Server
AWS Server
On-premise Server
```
Traffic Manager sabko manage kar sakta hai.

<br>

**Nested Endpoints**:

Ye Traffic Manager ke andar ek aur Traffic Manager profile hoti hai.

Isko nested routing bolte hain.

Example architecture:
```
Traffic Manager Global
        |
 -------------------------
 |                       |
TM Profile US        TM Profile Europe
```

Use case:
- Large enterprise architecture.

<br>

### Health Monitoring (Endpoint Monitoring)

Traffic Manager continuously check karta hai:
- Endpoint healthy hai ya down hai.

Iske liye health probes use hoti hain.

**Health Check ka Process**:

Traffic Manager time time par request bhejta hai.

Example:
```
GET https://webapp.com/health
```

Agar response aata hai:
```
200 OK
```
To endpoint healthy.

Agar response nahi aata:
```
Timeout
500 error
```
To endpoint unhealthy.

**Example**:

Servers:
```
East US
Europe
Asia
```

Agar:
```
Europe server down
```
Traffic Manager automatically Europe endpoint ko remove kar dega routing se.

<br>
<br>

### DNS Configuration

Traffic Manager DNS based service hai.

Iska apna DNS name hota hai:
```
myapp.trafficmanager.net
```
Company isko CNAME kar deti hai:
```
www.myapp.com → myapp.trafficmanager.net
```

Request flow:
```
User → DNS Query
DNS → Traffic Manager
Traffic Manager → Best endpoint
User → Direct endpoint
```

<br>
<br>

### TTL (Time To Live)

TTL ka full form hota hai:
```
Time To Live
```

DNS TTL decide karta hai ki DNS response kitni der tak cache mein store rahega.

Matlab:
- DNS resolver ko Traffic Manager se mila hua DNS response kitni der tak cache mein rakhna hai.

**Is case mein DNS Resolver kaha hota hai?**:

DNS resolver normally user ke network side par hota hai, Traffic Manager ke paas nahi.

Yeh usually in jagahon par hota hai:
- ISP DNS Resolver.
```
Airtel DNS
Jio DNS
BSNL DNS
```

- Public DNS Resolver:
```
Google DNS → 8.8.8.8
Cloudflare DNS → 1.1.1.1
OpenDNS → 208.67.222.222
```  

- Enterprise DNS Resolver:
```
Corporate DNS server
Active Directory DNS
```

<br>

TTL decide karta hai:
```
DNS response kitni der tak cache rahega.
```

Azure Traffic Manager DNS response ke saath TTL bhi return karta hai.

Example:
```
myshop.trafficmanager.net
→ 20.40.60.70
TTL = 60 seconds
```

Example:

Maan lo DNS response mila:
```
www.myshop.com → 20.40.60.70
TTL = 300 seconds
```
Iska matlab:
```
300 seconds = 5 minutes
```

Agle 5 minutes tak DNS resolver ko Azure Traffic Manager ke paas dubara DNS query nahi karni padegi.

Woh cached IP use karega.

Matlab:
- DNS 30 seconds tak same endpoint use karega.

<br>

**Important Reason**:

Traffic Manager ko fast failover karna hota hai.

Example:

Servers:
```
Mumbai
Amsterdam
Virginia
```

Agar Mumbai server down ho gaya aur TTL 1 hour ho:
- Users 1 hour tak down server ko hit karte rahenge aur user ko site down milegi.
- Isliye Traffic Manager low TTL recommend karta hai.

Example:
```
TTL = 30 seconds
```
Ab DNS jaldi update ho sakta hai.

<br>
<br>

### Traffic Manager Name Server

Traffic Manager ke DNS servers world wide distributed hote hain.

Azure Traffic Manager ka Name Server ek special DNS server hota hai jo decide karta hai ki user ko kis endpoint (server / region) ki IP address milegi.

<br>

**Normal DNS mein Name Server kya hota hai?**:

Internet mein Name Server ka kaam hota hai domain name ko IP address mein convert karna.

Example:

User browser mein type karta hai:
```
www.myapp.com
```

Browser ko actual mein IP address chahiye hota hai jaise:
```
20.50.10.30
```
Is conversion ko DNS resolution bolte hain.

<br>

Traffic Manager ke case mein Traffic Manager ka apna Name Server hota hai, jiska kaam hota hai domain name ko IP mein convert karna.

Example:

Tumne Traffic Manager profile banayi:
```
myapp.trafficmanager.net
```
Iske saath Azure automatically 4 authoritative name servers create karta hai.

Example:
```
ns1-01.azure-dns.com
ns2-01.azure-dns.net
ns3-01.azure-dns.org
ns4-01.azure-dns.info
```
Ye hi Traffic Manager ke Name Servers hote hain.

<br>

**Traffic Manager Name Server kya kaam karta hai?**:

Traffic Manager Name Server ka kaam hai, domain name ko ip mein convert karna.

Name Server directly conversion nahi karta, balki DNS record ke through domain name ka corresponding IP address batata hai.

Matlab:
- Name server ke paas ek database (DNS records) hota hai jisme ```domain → IP mapping``` stored hoti hai.

Example:
| Domain Name   |       IP        |
| ------------- | --------------- |
| myapp.trafficmanager.net |  192.168.50.10  |
| shop.trafficmanager.net  | 32.64.10.10     |

Name Server ka kaam hota hai batana ki is domain ka IP kya hai.

So:
```
www.example.com  →  93.184.216.34
```

Is mapping ko DNS record kehte hain.

<br>

**Example Flow**:

1 - DNS query receive karna:

Jab koi user request karta hai:
```
www.myapp.com
```

2 - Routing method apply karna:

Traffic Manager check karta hai ki tumne kaunsa routing method use kiya hai:
```
Priority
Weighted
Performance
Geographic
Multi value
Subnet
```

Example:
```
Performance routing
```

3 - To Traffic Manager check karega:
```
User kaha se request kar raha hai?
```

Example:
| User Location | Selected Server |
| ------------- | --------------- |
| India         | Mumbai          |
| Germany       | Europe          |
| USA           | East US         |


4 - Endpoint health check karna:

Traffic Manager continuously endpoints check karta hai.

Example:
```
https://mumbai-app.com/health
```
Agar endpoint unhealthy ho:
```
Traffic Manager usko DNS response mein nahi dega
```

5 - Correct IP address return karna:

Ab Name Server DNS response bhejega.

Example:

User India se request kar raha hai.

Traffic Manager response:
```
mumbai-app.azurewebsites.net
→ 20.40.60.70
```
Ab user directly Mumbai server par request bhej dega.

## Traffic Manager Routing Rules

Azure Traffic Manager mein multiple routing algorithms hote hain:
- Performance Routing.
- Priority Routing.
- Weighted Routing.
- Geographic Routing.
- MultiValue Routing.
- Subnet Routing.

<br>

### Performance Routing Algorithm

Performance routing ka purpose hai - User ko sabse lowest latency wala endpoint dena.

Yani jo server user ke sabse nearest ho network latency ke hisaab se, Traffic Manager usi ka address return karega.

Important baat:
- Ye geographic distance par based nahi hota, balki network latency mapping par based hota hai.

Azure continuously measure karta hai:
- internet latency.
- region connectivity.
- network performance

Iske basis par Traffic Manager decide karta hai kaunsa endpoint fastest hoga.

<br>

**Example Scenario**:

Ek global e-commerce company hai:

Servers:
```
East US
West Europe
Southeast Asia
```
Aur website hai:
```
www.globalshop.com
```
Traffic Manager configuration:

Routing Method = Performance

<br>

**Request Flow Example**:

Case 1 — User India se request karta hai

User browser mein open karta hai:
```
www.globalshop.com
```

DNS resolve hota hai:
```
globalshop.trafficmanager.net
```

Traffic Manager check karta hai:
```
User Location → India
```

Traffic Manager latency table check karta hai:
```
India → East US = 220 ms
India → West Europe = 180 ms
India → Southeast Asia = 40 ms
```

Traffic Manager choose karega:
```
Southeast Asia Endpoint
```

DNS response:
```
asia-server.azurewebsites.net
```

<br>

**Real DevOps Use Case**:

Performance routing best hai:
- Global applications.
- Gaming platforms.
- Streaming servicesSaaS platforms

Example companies:
- Netflix.
- Amazon.
- Spotify

Ye sab nearest region serve karte hain.

<br>
<br>

### Priority Routing Algorithm

Priority routing ka purpose hai: Primary server use karo aur agar wo fail ho jaye to backup server use karo.

Isko Failover routing bhi bolte hain.

Yahan har endpoint ko priority number diya jata hai.

Example:
```
Priority 1 → Primary server
Priority 2 → Secondary server
Priority 3 → Disaster recovery server
```
Traffic Manager lowest priority number wala endpoint choose karta hai.

<br>

**Example Scenario**

Company ke paas 3 servers hain:
```
Primary → East US
Secondary → West Europe
DR → Southeast Asia
```

Configuration:
```
East US → Priority 1
West Europe → Priority 2
Asia → Priority 3
```

<br>

**Normal Case**:

User request karta hai:
```
www.mybank.com
```
Traffic Manager check karega:
```
East US healthy?
```
Agar healthy hai:
```
User → East US
```

<br>

**Failure Case**:

Agar East US server crash ho gaya:

Traffic Manager health probe detect karega.
```
East US → Unhealthy
```
Ab routing change hogi:
```
User → West Europe
```

<br>

**Real World Example**:

Banking system.

Primary datacenter:
```
Mumbai
```
Backup datacenter:
```
Singapore
```
DR datacenter:
```
London
```

Routing:
```
Priority 1 → Mumbai
Priority 2 → Singapore
Priority 3 → London
```

Normal situation:
```
Users → Mumbai
```
Disaster situation:
```
Users → Singapore
```

<br>
<br>

### Weighted Routing Algorithm

Weighted routing ka use hota hai: Traffic ko percentage basis par distribute karne ke liye.

Har endpoint ko ek weight value diya jata hai.

Jis server ke paas jitna percent weight hoga, us server ke paas utne percent hi traffic jayega.

Example:
```
Server A → Weight 80
Server B → Weight 20
```

Traffic distribution:
```
80% → Server A
20% → Server B
```

<br>

**Example Scenario**:

Company new version test kar rahi hai.

Servers:
```
Old version server
New version server
```

Configuration:
```
Old version → Weight 90
New version → Weight 10
```

Result:
```
90% users → Old version
10% users → New version
```

<br>
<br>

### Geographic Routing Algorithm

Geographic routing ka purpose hai: User ki geographic location ke basis par endpoint select karna.

Example:
```
India users → Asia server
Europe users → Europe server
US users → US server
```

<br>

**Example Scenario**:

Video streaming platform.

Servers:
```
US
Europe
Asia
```

Configuration:
```
North America Users→ US server
Europe Users → Europe server
Asia Users → Asia server
```

**Example Flow**:

User India se request karta hai.

Traffic Manager detect karega:
```
User Country → India
```

Rule:
```
India → Asia endpoint
```

DNS response:
```
asia.streaming.com
```

<br>

**Real World Use Case**:

Geographic routing useful hota hai:

Legal Compliance

Example:
```
European users ka data Europe mein hi store hona chahiye.
```

Example law:
```
GDPR
```

Routing rule:
```
Europe users → Europe datacenter
Localization
```

Example:
```
India → Hindi website
Japan → Japanese website
France → French website
```

<br>
<br>

### MultiValue Routing Algorithm

MultiValue routing thoda different hai. Yahan Traffic Manager ek endpoint nahi return karta.

Instead:

Multiple endpoints DNS response mein return karta hai.

**Example**:

DNS response:
```
Server1 IP
Server2 IP
Server3 IP
```
Client khud choose karta hai.

<br>

**Example Scenario**:

DNS response:
```
10.0.1.1
10.0.1.2
10.0.1.3
```
Client randomly ek choose karega.

<br>

**Real Use Case**:

Ye mostly use hota hai:
- DNS-based load balancing.
- Stateless services.
- CDN style systems.

<br>
<br>

### Subnet Routing Algorithm

Subnet routing use hota hai: IP range ke basis par routing karne ke liye.

Example:
```
Office network → Private server
Public internet → Public server
```

Example Configuration
```
10.1.0.0/16 → Internal server
0.0.0.0/0 → Public server
```

**Example Flow**:

User request karta hai:
```
IP → 10.1.5.20
```

Traffic Manager rule check karega.
```
10.1.0.0/16 match
```

Routing:
```
Internal server
```

<br>

**Real DevOps Use Case**:

Corporate environments.

Example:

Company employees:
```
Office network
```

Routing:
```
Employees → Internal portal
```

Public users:
```
Public portal
```

<br>
<br>

### Simple Summary

Traffic Manager Routing Algorithms:
| Algorithm   | Purpose                         |
| ----------- | ------------------------------- |
| Performance | Nearest / fastest endpoint      |
| Priority    | Failover system                 |
| Weighted    | Traffic percentage distribution |
| Geographic  | Location based routing          |
| MultiValue  | Multiple endpoints return       |
| Subnet      | IP range based routing          |
