# LAB: Implementation of Azure Application Gateway

Ab Hum ek End-to-End project karenge Azure Application Gateway ke uper.

<br>

Ye ek Azure Application Gateway ka ek complete end-to-end real world project hai jisme:
- Ek VM banegi jisme Java application deploy hogi.
- Ek Azure App Service banega jisme Python application deploy hogi.
- Azure Application Gateway (Layer-7 load balancer) banega jo path ke basis par traffic route karega.

Routing rule example:
```
https://myapp.com/java   -> Java app on VM
https://myapp.com/python -> Python app on App Service
```

Isme SSL ko bhi handle karenge. Matlab SSL ka use bhi karenge.

Jab tum Azure Application Gateway ka URL ya Public IP browser mein hit karoge aur agar tum usme ```/java``` ya ```/python``` use karoge to request java app aur python app par jayegi, path ke according.

<br>
<br>

### Pehle Architecture Samajh Lo (Project Overview)

Hum jo architecture banane wale hain usme ye components honge:
```
Internet User
     │
     ▼
Azure Application Gateway
     │
 ┌───────────────┬────────────────┐
 │               │                │
 ▼               ▼
/java path       /python path
 │               │
 ▼               ▼
Java App (VM)    Python App (App Service)
```

**Simple flow**:
- User browser se request karega
```
http://public-ip/java
```

- Request Application Gateway par aayegi.
- Gateway URL ko inspect karega.
```
Path = /java
```
- Gateway request ko VM backend pool mein bhej dega, jaha pe java app chal rhi hai.
- Agar request aaye.
```
http://public-ip/python
```
- to gateway request App Service backend pool mein bhej dega, jaha python app chal rhi hai.

### Step 2 — Virtual Network Create karo

Application Gateway ko VNet ke andar hi deploy karna padta hai. Isliye VNet zaroori hai.

Azure Application Gateway ko bina Virtual Network (VNet) ke create karna mumkin hi nahi hai. Ye iska ek mandatory requirement hai. Agar aap ise kisi VNet ke bahar ya bina subnet ke deploy karne ki koshish karenge, to Azure Portal ya CLI aapko error de dega aur deployment fail ho jayegi.

Iske piche ke kuch bade technical kaaran ye hain:
- Dedicated Subnet ki Zaroorat:
  - Application Gateway ko hamesha ek dedicated subnet chahiye hota hai. Is subnet mein sirf Application Gateway ke instances hi reh sakte hain (kuch exceptions ko chhod kar).
  - Kyun? Kyunki Gateway ko backend servers (VMs, App Services) se communicate karne ke liye ek internal IP address ki zaroorat hoti hai jo usi VNet se aata hai.

- Backend Connectivity:
  - Application Gateway ek "Reverse Proxy" ki tarah kaam karta hai.
  - Jab traffic aata hai, to Gateway use receive karta hai aur phir naye connection ke zariye backend tak pahunchata hai.
  - Agar Gateway VNet mein nahi hoga, to wo aapke private VMs ya databases (jo VNet ke andar hain) se safely connect nahi kar payega.
 
<br>

Agar mai apne app ko azure app service pe create karu aur usko application gateway mein backend par rakhu, to isme to app service ko kisi vnet ki jarurat nhi hai to kya fir bhi application gateway ke liye vnet banana padega?

Haan, bilkul. Application Gateway ke liye VNet banana aur use ek dedicated subnet mein rakhna bilkul zaroorat (mandatory) hai, chahe aapka backend (App Service) kisi VNet mein ho ya na ho.

- Application Gateway ki apni Identity:
  - Application Gateway ek "Infrastructure-as-a-Service" (IaaS) based managed service hai. Iska matlab hai ki Azure piche backend par VMs deploy karta hai jinhe ek Internal IP address chahiye hota hai. Wo IP address sirf ek VNet ke subnet se hi mil sakta hai. Bina VNet ke, Application Gateway ke paas koi "ghar" nahi hoga jahan wo install ho sake.
 
- App Service (Backend) ka Connection:
  - Aapne sahi kaha ki App Service ko Gateway ke piche rakhne ke liye App Service ka VNet mein hona zaroori nahi hai (agar aap default setting use kar rahe hain).
  - Application Gateway Public URL (jaise myapp.azurewebsites.net) ke zariye App Service se baat kar sakta hai.
  - Lekin Gateway ko khud chalne ke liye aur internet se traffic lene ke liye ek VNet environment chahiye hi chahiye.
 
<br>

Create karo:
- Virtual Network.

Example configuration:
```
Name : appgw-vnet

```

Subnets:
```

```

Important rule:
- Application Gateway ko dedicated subnet chahiye.

**Python Web App Code**:

```

```

File:
```

```

Code:
```
from flask import Flask
app = Flask(__name__)

@app.route("/")
def home():
    return "Hello from Python App Service"

if __name__ == "__main__":
    app.run()
```

Ek ```requirements.txt``` naam se file create karo, jo jo required library hain vo likho, jaise:
```
flask
gunicorn
```

Ye ```gunicorn``` package isliye kyuki, Azure App Service (Linux) par jab aap custom Python app deploy karte hain, toh Azure ko batana padta hai ki application start kaise hogi. 

Jab aap development mein ```python app.py``` command chalate hain, toh Flask apna ek inbuilt server start karta hai. Lekin wo server sirf development (testing) ke liye hota hai. Wo ek waqt par sirf ek request handle kar sakta hai—bilkul waise hi jaise ek dukaan par sirf ek counter ho. Agar 100 log ek saath aa jayein, toh wo crash ho jayega.

Gunicorn wahan kaam aata hai. Ye ek WSGI (Web Server Gateway Interface) HTTP Server hai.

Jab bhi aap flask ke application ko production mein use kare to apko hamesha ```requirements.txt``` file mein ```gunicorn``` library jarur likhi hai, kyuki jab aap app service mein flask ke app ko deploy karte ho to app service gunicorn web server as a library install to bhi kar lega, To jab aap ```gunicorn --bind=0.0.0.0:8000 app:app``` command doge as startup command to flask app run ho jayegi.

- Wha se FTP credentials leke FTP tool se connect karlo, aur zip file ko ```wwwroot``` path pe copy kardo.

### Final Request Flow

Ab samajhte hain backend mein request kaise travel karti hai.

User browser mein likhta hai:
```
http://appgw-public-ip/java
```

**Step 1 — DNS Resolve**:

Browser DNS resolve karega.
```
appgw-public-ip
```
Azure ke Edge POP ko milega.

**Step 2 — Request Application Gateway par aati hai**:

Application Gateway ek Layer-7 Reverse Proxy hai. Ab request application gateway par aati hai.

Ye inspect karta hai:
```
URL
Headers
Cookies
Host
Path
```

**Step 3 — Path Match hota hai**:

Gateway dekhta hai
```
/java
```

Routing rule check hota hai.
```
/java → java backend
```

**Step 4 — Backend Pool select hota hai**:

Gateway backend pool select karega:
```
java-vm
```

**Step 5 — Gateway request forward karega**:

Gateway VM ko request bhejega
```
http://vm-private-ip:8080
```

**Step 6 — Java server response bhejega**:

Response:
```
Hello from Java VM Application
```

<br>

**Step 7 — Gateway response client ko bhejega**:

Gateway response ko forward karega
```
Client Browser
```
